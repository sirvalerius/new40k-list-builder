# Rules & data enforcement audit — 2026-06-22

Autonomous systematic audit of the rule areas that had been collecting frequent single bug
reports: **enhancement keyword requirements, weapon-option limit interpretation, optional
equipment requirements, stratagem applicability, and detachment rules.** Each fix is a single
atomic commit (listed with its hash) so it can be reviewed and rolled back independently.

All commits are on `main`; 80 unit tests pass and the production build is clean after each.

---

## Fixes applied

### 1. Enhancement requirement "X **unit** only" was not enforced — `4f790de`
- **Finding:** the keyword-requirement check added the day before only matched `"<KW> model
  only"`. **160 enhancements** — mostly the `(Upgrade)` ones that go on non-Character units —
  gate on `"<KW> unit only"` and were therefore assignable to *any* unit.
- **Fix:** `enhancementAllowed()` now matches both `MODEL(S) ONLY` and `UNIT(S) ONLY`.
- **Test:** Detachment with an Upgrade enhancement like *Camouflaged Snipers* (RANGERS unit
  only) — it now appears only on Rangers, not on other units.

### 2. Apostrophe-blind matching + narrow requirement parser — `ef65c63`
- **Finding:** three distinct gaps left **~238 requirements** unverified:
  - **Curly apostrophes.** "Emperor's Children" in the data uses `’`; the matcher compared
    against a straight `'`, so every `EMPEROR’S CHILDREN` / `T’AU` requirement silently passed.
  - **`<unit name> only`** with no "model/unit" word (e.g. *"Biologus Putrifier only."*,
    *"Great Unclean One only."*) was not recognised as a restriction at all.
  - **`(excluding …)` parentheticals** and **comma-OR lists** ("CANONESS, PALATINE or
    MINISTORUM PRIEST model only") broke the capture.
- **Fix:** fold `’ ‘ \`` → `'` in both `stratagemAppliesTo` and `enhancementAllowed`; strip
  parentheticals; allow the "model/unit" word to be optional; accept comma-separated OR lists;
  anchor the requirement to the opening clause so an "only" in effect text can't false-trigger.
- **Test:** a Death Guard plague-character enhancement ("Malignant Plaguecaster only") now only
  offers on that character; an Emperor's Children "Emperor's Children model only" enhancement
  now correctly refuses non-EC characters.

### 3. "For every N models, up to K …" lost the K multiplier — `da4c08b`
- **Finding:** **31 weapon options** read *"For every 5 models, up to K models can each have
  their X replaced"* but were capped at `floor(count/5)` (i.e. K=1), under-counting special
  weapons for Chosen, Plague Marines, Chaos Terminators, Raptors, etc. A 10-model Chosen squad
  allowed 2 special weapons instead of the correct 4.
- **Fix:** `parse_limit` captures K as `per`; `optionMax` returns `per × floor(count/N)`.
- **Test:** add a 10-model Chosen squad → the special-weapon option now allows **4** (2 per 5).

### 4. Single named-leader swaps parsed as "all" — `02526d5`
- **Finding:** **3 datasheets** (Dire Avenger Exarch additional catapult, Reiver Sergeant,
  Vespid Strain Leader) had a single-model conditional swap parsed as `all` (one per model),
  letting the option be taken up to the unit size.
- **Fix:** `parse_limit` treats a line naming a single leader model (sergeant/exarch/champion/
  …master/strain leader), with no "any number / for every / each / up to" wording, as fixed-1.
- **Test:** Reiver Squad → the Sergeant's bolt-carbine swap is now a single toggle, not N.

### 5. Constraint-note lines carried spurious base/grants — `fb87bfb` (data hygiene)
- **Finding:** **17 `note` options** ("* X cannot be replaced") had a garbled `base` and a
  `grants` entry that the note text happened to match. This is the same class of defect that
  hid the Sternguard bolt rifle (already fixed at runtime in `equippedWeapons`); this commit
  cleans it at the source so no other code path can be misled.
- **Fix:** the build clears `base`/`grants`/`model` on `note` options.
- **Test:** the Sternguard Veteran Squad card shows its base bolt-rifle profile; no datasheet
  hides a base weapon.

## Areas audited and verified sound (no change needed)

- **Stratagem applicability.** The apparent "unrecognised target keyword" hits are effect-text
  noise ("ONLY THAT ENEMY", "CRITICAL HIT") or phrases like "ONE HERETIC ASTARTES CHARACTER"
  that already match by substring on the in-vocab `HERETIC ASTARTES`. The apostrophe fix (#2)
  also tightens stratagems for `Emperor's Children` / `T'au`.
- **Detachment exclusivity.** `exclusive_tag` is enforced (two detachments sharing a tag →
  `DET_TAG` error). Verified for the only tag used twice (`ACROBATIC`).
- **Chapter-lock restrictions.** The 10 SM "can include <Chapter> but not other ADEPTUS
  ASTARTES" detachments are enforced via Chapter binding + the roster filter (hides other
  Chapters' units, keeps generic Adeptus Astartes, applies the Black Templars psyker exclusion).
- **`is_upgrade` flag.** 0 mismatches against the canonical `(Upgrade)` name tag.
- **Ally caps.** Per-battle-size caps (`cap_incursion/strike/onslaught`) read and summed against
  allied points.
- **Crisis "slots".** 110 "up to K of the following" options parse to `slots` with a
  `modelCount × K` group cap.
- **Paid wargear.** No orphan costs — every paid option is surfaced as a pickable line.

## Validation: requirement enforcement does not over-block

A Python mirror of the matcher was run against every faction: of **935 enhancements, 928 parse a
keyword requirement, and only 3 match zero units** — all *"Astra Militarum Titanic **Character**
model only"* in the Steel Hammer detachment. AM has 19 Titanic units but none are Characters, so
these are genuinely unit-less in the current datasheet set (the detachment expects a super-heavy
commander that isn't present), **not** a parsing error. Confirms the enforcement is accurate, not
over-aggressive. No code change — forcing super-heavies to `is_character` would be wrong.

## Known minor / deferred
- **Replaced-weapon display on ~84 swap options** (mostly Forge World / Legends: Relic
  Contemptor, Fire Raptor, Storm Eagle, Thunderhawk; plus 3 garbled bases on Sanctifiers /
  Wolf Scouts). The option's `base` weapon name doesn't string-match the datasheet's weapon
  profile (plural vs singular, a leading count like "2 twin hellstrike missile launchers", or a
  different name), so the replaced weapon keeps showing alongside the new one. **Display-only —
  points and legality are unaffected.** Fixing needs fuzzy base-matching (regression-prone), so
  it's deferred rather than changed unattended.
- Enhancement requirements fail **open**: if a required keyword isn't in the faction vocab the
  enhancement is allowed rather than wrongly blocked (safe default).
- Leader vs Support typing isn't in the source data, so the attachment cap is "≤2 Characters
  per bodyguard" rather than strictly "exactly 1 Leader + 1 Support".
- "(excluding X)" exclusions on enhancements are dropped from the requirement (the main
  keyword is still enforced); exclusions are not separately checked.
